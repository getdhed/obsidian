# 📌 Таблицы-связки (Many-to-Many) в PostgreSQL

## 🔹 Что такое таблица-связка?

Таблица-связка (junction table, pivot table) — это промежуточная таблица, которая используется для реализации связи **многие-ко-многим (Many-to-Many)** между двумя сущностями.

Пример из предметной области:

- Один фильм → много актёров
- Один актёр → много фильмов

Обычным `FOREIGN KEY` такую связь реализовать нельзя, поэтому создаётся отдельная таблица.

---

# 🎬 Пример: фильм и актёр

## Основные таблицы

```sql
CREATE TABLE films (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL
);

CREATE TABLE actors (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL
);
```

## Таблица-связка

```sql
CREATE TABLE film_actors (
    film_id INT NOT NULL REFERENCES films(id) ON DELETE CASCADE,
    actor_id INT NOT NULL REFERENCES actors(id) ON DELETE CASCADE,
    PRIMARY KEY (film_id, actor_id)
);
```

---

# 🔹 Почему используется составной PRIMARY KEY?

```sql
PRIMARY KEY (film_id, actor_id)
```

### Это нужно для:

1. ✅ Запрета дубликатов
   
   Нельзя добавить одного и того же актёра к одному фильму дважды.

2. 🚀 Автоматического создания индекса
   
   Ускоряет JOIN-запросы.

3. 🧠 Логической целостности данных


Без составного первичного ключа можно было бы вставить:

```sql
INSERT INTO film_actors VALUES (1, 5);
INSERT INTO film_actors VALUES (1, 5);
```

И база это разрешила бы.

---

# 🔹 Почему не делать отдельный id?

Можно сделать так:

```sql
id SERIAL PRIMARY KEY
```

Но тогда обязательно нужно добавить:

```sql
UNIQUE (film_id, actor_id)
```

Иначе будут возможны дубликаты.

Поэтому в таблицах-связках обычно **не создают отдельный id**, а используют составной PRIMARY KEY.

---

# 🔹 ON DELETE CASCADE — зачем?

```sql
REFERENCES films(id) ON DELETE CASCADE
```

Если удалить фильм, все его связи автоматически удалятся из `film_actors`.

Это предотвращает появление "висячих" записей.

Без CASCADE пришлось бы вручную сначала удалять связи.

---

# 🔹 Как это выглядит в данных

film_actors:

| film_id | actor_id |
|----------|-----------|
| 1        | 3         |
| 1        | 7         |
| 2        | 3         |

Это означает:

- Фильм 1 → актёры 3 и 7
- Фильм 2 → актёр 3

---

# 🔹 Как делать JOIN

```sql
SELECT f.name, a.name
FROM films f
JOIN film_actors fa ON f.id = fa.film_id
JOIN actors a ON a.id = fa.actor_id;
```

---

# 📌 Когда нужна таблица-связка?

Когда выполняются оба условия:

- A связано с многими B
- B связано с многими A

Примеры:

- Пользователи ↔ Роли
- Студенты ↔ Курсы
- Заказы ↔ Товары
- Фильмы ↔ Актёры

---

# 🎯 Краткое правило

Если видишь связь "многие-ко-многим" — создавай отдельную таблицу с двумя FOREIGN KEY и составным PRIMARY KEY.

---

# 🚀 Продвинутая версия

Если нужно хранить дополнительную информацию о связи, например:

- роль актёра в фильме
- гонорар

Тогда таблица-связка расширяется:

```sql
CREATE TABLE film_actors (
    film_id INT REFERENCES films(id) ON DELETE CASCADE,
    actor_id INT REFERENCES actors(id) ON DELETE CASCADE,
    role_name TEXT,
    fee NUMERIC,
    PRIMARY KEY (film_id, actor_id)
);
```

---

# 🧠 Вывод

Таблица-связка:

- реализует Many-to-Many
- защищает от дубликатов
- обеспечивает целостность данных
- ускоряет JOIN
- масштабируется при необходимости

Это базовый и обязательный паттерн проектирования реляционных БД.
