# 📦 Protocol Buffers (Protobuf)

## Что такое Protobuf

Protocol Buffers — это бинарный формат сериализации данных, разработанный Google.

Главная идея:

> Передавать структурированные данные быстрее и компактнее, чем JSON.

Protobuf часто используется вместе с:

- [[gRPC]]
    
- микросервисами
    
- внутренними API
    

---

# 🧠 Почему не JSON?

JSON:

- текстовый
    
- удобный для человека
    
- но большой по размеру
    
- медленнее парсится
    

Protobuf:

- бинарный
    
- компактный
    
- быстрее сериализуется
    
- строго типизирован
    

---

# 🧱 Как работает Protobuf

Вместо передачи структуры в JSON, ты сначала описываешь её в `.proto` файле.

Пример:

syntax = "proto3";  
  
message User {  
  int32 id = 1;  
  string name = 2;  
  string email = 3;  
}

Это называется:

> Schema (схема данных)

---

# 🔢 Что означают номера полей

int32 id = 1;

Число `1` — это номер поля.

В бинарном формате данные кодируются по номеру, а не по имени.

Это делает формат компактным.

Важно:

> Номер поля нельзя менять после выпуска API.

---

# ⚙ Генерация кода

Из `.proto` файла генерируется код для нужного языка.

Для Go:

protoc --go_out=. user.proto

Создаётся Go-структура.

---

# 🧩 Пример использования в Go

После генерации:

user := &pb.User{  
    Id:    1,  
    Name:  "Nazar",  
    Email: "test@mail.com",  
}

Сериализация:

data, err := proto.Marshal(user)

Десериализация:

var user pb.User  
err := proto.Unmarshal(data, &user)

---

# 🚀 Protobuf + gRPC

Protobuf чаще всего используется с [[gRPC]].

В `.proto` можно описывать не только структуры, но и сервисы:

service UserService {  
  rpc GetUser(UserRequest) returns (UserResponse);  
}

Это уже:

- описание API
    
- контракт между сервисами
    

---

# 📦 Типы данных в Protobuf

- int32
    
- int64
    
- string
    
- bool
    
- bytes
    
- repeated (массив)
    
- message (вложенная структура)
    

Пример массива:

repeated string tags = 4;

---

# 🔄 JSON vs Protobuf

|JSON|Protobuf|
|---|---|
|Текст|Бинарный|
|Читаемый|Не читаемый|
|Гибкий|Строгая схема|
|Медленнее|Быстрее|
|Больше размер|Меньше размер|

---

# 🧠 Когда использовать Protobuf

Подходит для:

- микросервисов
    
- high-load систем
    
- внутреннего общения сервисов
    
- gRPC
    

Не подходит для:

- публичных API для фронтенда
    
- случаев, где важна читаемость
    

---

# 🔐 Версионирование

Protobuf поддерживает безопасное изменение схемы:

Можно:

- добавлять новые поля
    
- не использовать старые
    

Нельзя:

- менять номера полей
    
- менять тип существующего поля

# 📌 Главное понимание

Protobuf — это:

> Строго типизированный бинарный формат передачи данных по заранее определённой схеме.

Он нужен для:

- скорости
    
- компактности
    
- надёжного контракта между сервисами

REST + JSON — удобно для человека  
gRPC + Protobuf — удобно для машин
