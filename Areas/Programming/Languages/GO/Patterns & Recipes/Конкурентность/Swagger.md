
# Swagger (OpenAPI)

## Что такое Swagger?

**Swagger** — это инструмент для описания и документирования [[REST API]].

Фактически сегодня Swagger = реализация стандарта **OpenAPI Specification (OAS)**.

Он позволяет описать:

- эндпоинты
    
- HTTP-методы
    
- параметры
    
- формат запросов
    
- формат ответов
    
- статус-коды
    
- схемы данных
    

Swagger — это **контракт для [[API]]**.

---

## Зачем нужен Swagger?

Когда проект растёт, просто писать документацию в `.md` или Word становится неудобно.

Проблемы обычной документации:

- может устареть
    
- не связана с кодом
    
- нельзя протестировать запрос
    
- нельзя автоматически сгенерировать клиент
    

Swagger решает это:

- генерирует UI-документацию
    
- позволяет отправлять запросы из браузера
    
- может генерировать клиентский код
    
- может генерировать серверные заглушки
    
- формализует контракт между фронтом и бэком
    

---

## Swagger UI

**Swagger UI** — это веб-интерфейс, который показывает:

- список всех эндпоинтов
    
- описание параметров
    
- кнопки `Try it out`
    
- реальные ответы сервера
    

По сути — встроенный тестер вместо [[Postman]].

---

## Как это выглядит (пример OpenAPI)

paths:  
  /users:  
    get:  
      summary: Get users  
      parameters:  
        - name: limit  
          in: query  
          schema:  
            type: integer  
      responses:  
        200:  
          description: OK

Это уже не просто текст, а машиночитаемая схема.

## Swagger в Go

В Go часто используют:

- `swaggo/swag`
    
- `gin-swagger`
    
- `echo-swagger`
    

Пример аннотации:

// @Summary Get user  
// @Param id path int true "User ID"  
// @Success 200 {object} User  
// @Router /users/{id} [get]

После генерации появляется `/swagger/index.html`.

Связанные темы:

- [[HTTP]]
    
- [[Router]]
    
- [[Middleware]]
    
- [[REST API]]
    
- [[JSON]]
    
- [[Статус коды HTTP]]
    

---

## Важно понимать

Swagger — это:

- не протокол
    
- не замена [[HTTP]]
    
- не замена [[REST API]]
    

Это **описание API + инструменты вокруг него**.

---

## Когда использовать?

Подходит если:

- больше 5–10 эндпоинтов
    
- есть фронтенд
    
- есть мобильное приложение
    
- есть несколько сервисов ([[Microservices]])
    

В маленьком проекте можно обойтись `.md`,  
в серьёзном — без OpenAPI начинается хаос.
